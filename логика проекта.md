## Два уровня доступа

### Уровень 1 — Owner (Евгений)

- `settingSources: ["user", "project"]` — читает `~/.claude/CLAUDE.md` (глобальные инструкции)
- `allowedPaths`: широкие (/opt, /root, /home, /tmp...)
- Команды: все включая `/restart`, `/reloadbot`
- **Системный промпт** (`OWNER_SAFETY_PROMPT`): английский, базовые правила безопасного удаления файлов и ограничения путей
- **CLAUDE.md** (`workspace/CLAUDE.md`): очень богатый — FitCoach, Отклыкер, инфра-маршрутизация, граф памяти, маркетинговые книги, параллельные агенты, карта admin-операций

### Уровень 2 — Guest (все остальные)

- `settingSources: ["project"]` — не читает `~/.claude` владельца
- `allowedPaths`: только `/opt/vault/{userId}/` + `/tmp/telegram-bot`
- Команды: нет `/restart`, нет `/reloadbot`
- **Системный промпт** (`NEW_GUEST_SAFETY_PROMPT`): скиллы (`ls skills/` при каждом запросе), адаптация стиля под пользователя, сессии с темами, публичная страница `https://jinru.pro/u/{userId}/`, антипетля
- **CLAUDE.md**: генерируется из шаблона (`src/templates/guest-claude-md.ts`) — стандартный
- **Модель: deepseek-chat** (DeepSeek API, ключ `DEEPSEEK_API_KEY`; vision → gemini-2.5-flash)

**Исключение — Ксения (`893951298`):** тот же Guest-профиль, но модель `claude-sonnet-4-6` (осознанное решение). `allowedPaths` расширены — включают `CLAUDE_WORKING_DIR` (owner workspace, read-only в контексте изоляции).


## Sandbox-контейнеры (добавлено 2026-05-06)

### Модель изоляции

Каждый гостевой пользователь получает Docker-контейнер как среду исполнения кода. Claude при этом остаётся общим (HTTP-вызовы в OpenRouter/Anthropic) — контейнер не держит Claude-процесс.

```
Сообщение тестера
  → Бот (один процесс) → OpenRouter/DeepSeek (HTTP)
  ← DeepSeek: tool_call Bash("ls /workspace")
  → docker exec claude-user-{userId} bash -c "ls /workspace"  ← выполняется в контейнере
  ← результат → DeepSeek → ответ пользователю
```

Контейнер в покое: **~8-10 MB** (sleep infinity + kernel overhead). Тяжёлый процесс появляется только когда пользователь запускает что-то сам (сайт, скрипт).

### Образ контейнера: `claude-user-sandbox:latest`

Base: debian:bookworm-slim. Установлено: Bun, Node.js 20, Python 3, git, nginx, build-essential, sqlite3, базовые cli-инструменты. Образ собирается на сервере, 314 MB на диске.

### Монтирование внутри контейнера

| Хост | Контейнер | Права |
|---|---|---|
| Docker volume `claude-user-{userId}-data` | `/workspace` | RW — persistent хранилище пользователя |
| `/root/.claude/` | `/root/.claude/` | RO — credentials + settings для Claude CLI |
| `/var/lib/claude-bot/dropbox/{userId}/` | `/tmp/dropbox` | RW — file-drop для ask-user/send-file MCP |

### Лимиты

| Профиль | CPU | RAM | docker.sock |
|---|---|---|---|
| Owner (Евгений) | без лимита | без лимита | ✅ примонтирован + /var/log:ro |
| Guest | 1.0 ядро | 512 MB + 512 MB swap | ❌ |

### Жизненный цикл контейнера

`absent` → (первое сообщение) → `running` → (15 мин без сообщений) → `paused` → (24ч) → `stopped`

- `paused`: cgroup-freezer, RAM не освобождается, resume мгновенный
- `stopped`: RAM освобождается, workspace-volume сохраняется
- `remove`: только явным действием владельца

### Invite-only система

Новый пользователь пишет боту → бот отвечает «доступ закрыт» и сохраняет `PendingInvite` в `/var/lib/claude-bot/pending/{userId}.json` → Евгений получает Telegram-уведомление с кнопками **Одобрить / Отклонить** → при одобрении: пользователь добавляется в `system/users.json` как `new_guest` с `containerEnabled: true`, контейнер создаётся автоматически при первом сообщении.

### Ёмкость текущего сервера (1 vCPU, 1.9 GB RAM)

- Зарегистрировано: **20-30 тестеров** (большинство молчат)
- Одновременно активны (пишут боту): **5-7**
- Одновременно гоняют тяжёлый код (npm install, серверы): **2-3**
- Узкое место: **CPU (1 ядро)**, не RAM

При апгрейде до CX22 (4 vCPU, 8 GB, €7.49/мес): 50-100 активных тестеров.

### Файлы

| Файл | Что делает |
|---|---|
| `Dockerfile.user` | Образ sandbox-контейнера |
| `src/containers/manager.ts` | `ContainerManager` — lifecycle, exec, idle watchdog |

| `src/containers/paths.ts` | Пути: volumeName, containerName, dropboxDir и т.д. |

| `src/handlers/callback.ts` | invite_approve / invite_deny callbacks |
| `src/user-registry.ts` | `addUser()` — динамическое добавление в users.json |

### Подавление спама в чате (отдельная задача)

Два независимых источника «мусора», которые виде пользователь:

- **Промежуточные text-сегменты Claude** между tool-вызовами («сейчас гляну...», «вижу...»). Источник: `src/handlers/streaming.ts`, ветка `statusType === "text"`. Решение: либо удалять промежуточные сегменты на `done` (как сейчас удаляются `toolMessages`), либо буферизовать и публиковать только последний.
- **Технические ошибки и трейсы**, утекающие через `catch` блоки в хендлерах. Решение: маппинг типов ошибок → дружелюбные сообщения, детали — в audit log и stderr.

Обе фильтрации обязательны до миграции, чтобы новый сервер встретил пользователей чисто.

### Решение: прямой DeepSeek через Claude CLI

DeepSeek по адресу `https://api.deepseek.com/anthropic` работает с Claude Code CLI **только** при наличии полного набора env переменных переопределения моделей. Без них CLI шлёт `metadata.user_id` в формате, который DeepSeek отвергает (`Invalid 'user_id': pattern '^[a-zA-Z0-9_-]+$'`).

Полный набор env, который применяется в `deepseekEnv` (`src/config.ts`):

```
ANTHROPIC_API_KEY=<DEEPSEEK_API_KEY>
ANTHROPIC_BASE_URL=https://api.deepseek.com/anthropic
ANTHROPIC_DEFAULT_SONNET_MODEL=deepseek-chat
ANTHROPIC_DEFAULT_OPUS_MODEL=deepseek-reasoner
ANTHROPIC_DEFAULT_HAIKU_MODEL=deepseek-chat
ANTHROPIC_MODEL=deepseek-chat
```

С этим набором CLI понимает что это сторонний провайдер и переключается в режим без metadata. Тестировал — отвечает: `is_error: False, result: "Hi!", usage: 25618 in / 13 out`.

Vision (фото) DeepSeek не поддерживает — `mediaHint`-ветка в `session.ts` остаётся на OpenRouter Gemini. OpenRouter ключ читается из `process.env.OPENROUTER_API_KEY` с fallback на per-user `/opt/vault/{id}/openrouter-key.txt` (унифицировано в этой же сессии, см. `getNewGuestOpenRouterKey`).

## Что сделано — Раунд 1 Фазы A (2026-05-06)

Это первый пакет локальных правок по плану Фазы A (без касания сервера). Пять пунктов из шести выполнены.

### 1. Бот перестал «думать вслух» в чат

Раньше, пока Claude работал с инструментами, пользователь видел промежуточные мысли: «сейчас проверю», «нашёл, скачаю», «вижу вот что». Это сбивало с толку — особенно нетехнических людей. Выглядело как куча мусора перед финальным ответом.

Теперь все такие промежуточные сообщения **удаляются** сразу после завершения работы — остаётся только итоговый ответ. Во время выполнения пользователь по-прежнему видит прогресс (счётчик инструментов), а в конце чат остаётся чистым.

Правка в `src/handlers/streaming.ts`: промежуточные текстовые сегменты между вызовами инструментов теперь попадают в тот же список на удаление, что и служебные сообщения о статусе.

### 2. Технические ошибки больше не утекают в чат

В семи местах кода при сбое пользователь раньше получал сырую ошибку с трассировкой: `❌ Error: ReferenceError at ...` или похожее. Это пугает и не несёт никакой пользы обычному пользователю.

Теперь вместо этого — короткое человеческое сообщение: «не получилось обработать фото, попробуй ещё раз». Полная ошибка при этом уходит только в журнал сервера.

Создана общая вспомогательная функция `replyFriendly` в `src/utils.ts`. Она хранит разные тексты для разных ситуаций: обработка фото, транскрипция голоса, перезапуск службы, обработка документа и так далее. Вызвана в семи местах: `src/handlers/text.ts`, `audio.ts`, `voice.ts`, `callback.ts`, `media-group.ts`, `document.ts`, `commands.ts`.

### 3. Учёт токенов на каждого пользователя

Создан новый модуль `src/metering.ts`. Он ведёт базу данных `metering.sqlite` — лёгкую встроенную, ничего дополнительно ставить не нужно. Каждый запрос к моделям теперь записывается: кто пользователь, какая модель, сколько входящих и исходящих токенов, сколько из кэша, сколько это примерно стоило в долларах.

Источники запросов разделены по ярлыкам:
- `bot-anthropic` — Sonnet через Claude CLI (Евгений, Ксения)
- `bot-deepseek` — DeepSeek через Claude CLI (остальные гости)
- `bot-openrouter` — OpenRouter напрямую (фото через Gemini)

На будущее зарезервированы ярлыки для Open Design.

Цены прошиты в коде по моделям — потом можно сверить с реальными счетами. Перехват данных идёт в двух точках: `src/session.ts` (после ответа от Claude CLI) и `src/engines/openrouter.ts` (после ответа от OpenRouter).

Файл `metering.sqlite` добавлен в `.gitignore` — он содержит данные пользователей, не для git.

### 4. Claude теперь знает свои ресурсы и возможности

Это решение двух постоянных проблем:

**Проблема 1 — ресурсы.** Claude в гостевом контейнере не знал, что у него только 512 МБ памяти и одно ядро. Запускал тяжёлые сборки, упирался в потолок и падал, не понимая почему. Теперь в его инструкции есть блок «ТВОИ РЕСУРСЫ» с лимитами и советом: перед тяжёлой задачей проверить `free -m`/`df -h` и оценить, влезет ли.

**Проблема 2 — возможности.** Claude (особенно DeepSeek) регулярно отвечал «у меня нет доступа к интернету» или «я не могу установить пакет», хотя всё это есть. Теперь в инструкции есть блок «ТВОИ ВОЗМОЖНОСТИ»: явный список — веб-поиск и загрузка страниц есть, Bash с интернетом работает, любые менеджеры пакетов устанавливаются без вопросов, что уже предустановлено в контейнере. И прямой запрет: «никогда не отвечай "у меня нет доступа к интернету"».

Эти блоки добавлены в три системных инструкции в `src/config.ts` (`buildNewGuestSafetyPrompt`, `buildOwnerSafetyPrompt`, `buildGroupSystemPrompt`) и в шаблон файла памяти гостя (`src/templates/guest-claude-md.ts`). Claude видит эту информацию и в инструкции при запуске, и в файле памяти, который читается в начале каждой сессии.

### 5. Починен баг с приёмом нового пользователя

При попытке принять Марину (`188062855`) кнопкой «Одобрить» бот падал: она уже была в `users.json`, но без полной информации. Функция добавления пользователя не умела дополнять существующую запись — только добавлять новую, и на повторе крашилась.

Теперь функция работает по принципу «добавь или дополни»: если пользователь уже есть — обновляет его запись, не трогает существующие поля. Если нет — создаёт новую. При повторном нажатии «Одобрить» владелец видит «уже был одобрен ранее» вместо ошибки.

Правки в `src/user-registry.ts` и обработчике кнопки `invite_approve` в `src/handlers/callback.ts`.

## Что сделано — Раунд 2 Фазы A (2026-05-06)

### 1. Сбор метрик контейнера

Раньше бот никак не отслеживал, сколько ресурсов занимает контейнер конкретного пользователя. Лимиты были прописаны при запуске (512 МБ памяти, 1 ядро), но текущее фактическое состояние — нигде не считалось.

Создан новый файл `src/containers/metrics.ts` с двумя функциями. Первая снимает метрики одного пользователя: есть ли его контейнер, запущен ли, сколько занимает оперативная память, какова нагрузка на процессор, каков размер папки `/opt/vault/{userId}/`. Вторая снимает метрики всех пользователей сразу, параллельно.

Под капотом дёргаются стандартные команды Docker (`docker ps`, `docker stats`) и `du`. Если контейнер ещё не создан или Docker не установлен (как на локальном маке) — функции возвращают пустые значения, не падают.

### 2. Веб-сервер дашборда

В боте уже работал встроенный мини-сервер на порту 3847 — для обмена данными с часами Apple Watch. Рядом запущен второй на порту 3848. Файл `src/dashboard-server.ts`.

Что сервер обслуживает:
- `/` — страница-заглушка лендинга.
- `/dashboard` — страница дашборда пользователя.
- `POST /api/me` — приложение внутри Telegram присылает подписанную строчку от Telegram, сервер проверяет подпись (по токену бота) и возвращает данные именно этого пользователя: токены, ресурсы контейнера, тарифный ярлык, модель.
- `POST /api/admin/all` — то же самое, но только для владельца: возвращает сводку по всем пользователям.

Подпись проверяется по официальной инструкции Telegram. Если что-то не так — возвращается ошибка авторизации, никаких внутренних деталей наружу.
