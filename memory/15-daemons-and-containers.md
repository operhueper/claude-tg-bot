# Демоны и постоянные контейнеры

Зафиксировано 2026-05-08 после длительного обсуждения с владельцем.

## Контекст

Гость 946882308 (Гоша) держит свой Telegram-бот `koen-assistant` (BRITVA Assistant Bot, @koenassistant_bot, токен `8084841103:...`) как systemd-юнит на хосте `proboi-bot 89.167.125.175`, путь `/opt/vault/946882308/bots/assistant/bot.py`. Хочет рестартовать сам. Решили вместо этого перевести его бот **внутрь его гостевого контейнера** и сделать «постоянные автоматизации внутри контейнера» полноценной фичей платформы.

## Смена ментальной модели

- **Было:** контейнер = ephemeral workspace для Claude-сессии, паузится при простое.
- **Станет:** контейнер = персональный slot пользователя, живёт 24/7. Биллинг — это токены + always-on slot.
- На текущем этапе все гости — на платном тарифе бесплатно, поэтому различие Free/Paid пока неактивно. Заложить в архитектуру, но включать с тарифами потом.

## Решения по реализации

### Обнаружение демонов
**Решение:** определяем автоматически. Если внутри контейнера крутится долгоживущий процесс — это «автоматизация». Пользователю показываем «у тебя N автоматизаций из M доступных».

### Лимит автоматизаций
**Решение:** 3 на пользователя. Это будущий бейзлайн для тарифов (Базовый = 1, Профи = 3, Студия = 10 — цифры пока с потолка).

### Пересоздание контейнера при изменении spec
**Решение:** просто пересоздаём, демоны перезапускаются. Не блокируем пересоздание ради демонов — фича перезапуска должна быть надёжной.

### Restart policy на Docker
- `--restart=unless-stopped` для всех контейнеров.
- Контейнеры выживают перезагрузку хоста и рестарт docker daemon'а.
- Контейнеры переживают рестарт сервиса `claude-tg-bot` (они независимы).

### Внутренний надсмотрщик (PID 1)
- Заменяет `sleep infinity` в каждом контейнере.
- **Реализация — Go-бинарь** (~5-10 МБ, не Python — Python 30-50 МБ слишком жирно для 512 МБ тарифа).
- Функции: запуск зарегистрированных автоматизаций, авто-restart при падении, логи в `/opt/vault/<id>/logs/<daemon>.log`.
- Бинарь вкомпилен в образ `claude-user-sandbox:latest`, пользователь не может подменить.

### OOM-политика
**Решение:** 3 падения за 10 минут → стоп автоматизации, уведомление пользователю «твоя автоматизация падает, возможно нужен тариф побольше», уведомление владельцу.

### Сетевые ограничения исходящего

| Порт/протокол | Действие |
|---|---|
| Исходящий 25 (SMTP) | **Закрыт полностью** |
| Исходящий 465/587 (SMTP-auth) | Разрешён, **rate-limit 30 коннектов/час на пользователя** |
| HTTP/HTTPS | Открыт |
| Прочее | Открыто пока, обсудим при появлении абуза |

Логика: своя почта через Mailgun/SendGrid/Resend по HTTPS API — не наша зона ответственности (это пользователь, его аккаунт, его IP отправителя). Прямой SMTP с нашего IP — наша головная боль (RBL-листы), потому закрываем.

### Анти-майнинг и анти-скан
**Решение:** не банить автоматом. Мониторинг «средний CPU > 70% за час» → уведомление владельцу + уведомление пользователю «у тебя неестественное поведение, проверь автоматизацию». Дальше владелец смотрит руками.

Признаём: абузеры будут отнимать время вручную. На малых масштабах (<10 пользователей) это терпимо, при росте автоматизировать.

### Исходящий трафик
**Решение:** 20 ГБ/день на пользователя. Превышение — троттлинг до 100 КБ/с до утра, уведомления.

### Уведомления владельцу
**Решение:** канал в Телеграме «проблемы пробоя» (`https://t.me/+r-HtVcVXFAg2OWRi`). Все алерты (CPU-спайки, OOM, троттлинг трафика, abuse-флаги) идут туда. Email `abuse@proboi.site` владелец настроит сам позже.

Реализация: нужно добавить @proboiAI_bot админом в канал, бот заберёт chat_id из логов и сохранит в env. Функция `notifyOwnerProblemChannel(text)` рассылает в канал.

### Не-HTTP протоколы наружу
**Решение:** запрещены. Только HTTP/HTTPS наружу через стандартные порты.

## Свои сайты — финальная позиция (2026-05-08 уточнено)

**Никаких пользовательских сайтов на нашей инфраструктуре.** Ни поддоменов, ни внешних доменов с проксированием в контейнер.

Если пользователь хочет сайт — берёт обычный хостинг отдельно от нас. Платформа = только Telegram-ассистент + автоматизации в контейнере, никаких публичных HTTP-эндпоинтов пользователя.

Это снимает с владельца роль информационного посредника по 149-ФЗ для пользовательских сайтов и закрывает целый класс правовых рисков (фишинг, пиратка, реестр РКН).

Существующая статика на `proboi.site/u/<id>/` — пересмотреть, нужна ли вообще. Если оставляем — это всё ещё хостинг с нашей стороны и риски остаются. Возможно, выпилить в рамках этой же работы.

3. ✅ Go-надсмотрщик `scripts/daemon-runner/daemon-runner` (Linux/amd64, 2.2 MB, статика). PID 1 в `claude-user-sandbox:latest`.
4. ✅ Манифест `.daemons.yaml`, лимит 3 (в надсмотрщике), Claude управляет файлом по запросу пользователя.
5. ✅ OOM-политика: надсмотрщик пишет `<vault>/.daemons-events/*-crashloop.json` при 3+ падениях за 10 минут; TS-watcher `src/crashloop-watcher.ts` опрашивает раз в 30с и шлёт уведомления.
6. ✅ iptables (`scripts/firewall/`): блок исходящего 25, rate-limit hashlimit 30/час на 465/587, 20 GB/день egress + tc-throttle до 100 KB/s.
7. ✅ CPU-monitoring (`scripts/monitoring/`): docker stats каждую минуту, history 60 точек, триггер при avg >70% за час.
8. ✅ Миграция koen-assistant: systemd на хосте отключён, бот живёт внутри контейнера 946882308 под надсмотрщиком, polling работает, apscheduler тикает.
9. ✅ Pause/stop контейнера skip'аются если `.daemons.yaml` содержит `enabled: true` (не паузим контейнеры с автоматизациями).
