# Логика бота — карта проекта на человеческом языке

## 1. Что делает бот

Telegram-бот, который позволяет общаться с Claude (и DeepSeek) через мессенджер: текстом, голосом, фото, документами. У каждого пользователя своя долгосрочная память, набор инструментов и изолированная рабочая среда. Бот живёт на сервере, думает сам, исполняет код, работает с Google-документами и отправляет файлы обратно в чат.

---

## 2. Кто пользуется и в каких ролях

Есть два типа пользователей: **владелец** и **гость**.

**Владелец** — Евгений (числится под ID 292228713). Он имеет полный доступ: может управлять файлами бота, перезапускать сервис командой `/restart`, видеть любые логи, работать с широким списком папок на сервере. Сейчас и владелец, и все гости работают на одной языковой модели — DeepSeek (`deepseek-chat` для обычных задач, `deepseek-reasoner` для тяжёлых). Это решение принято для экономии: DeepSeek в десятки раз дешевле Anthropic при сравнимом качестве.

**Гость** — все остальные одобренные пользователи. Каждый гость работает внутри своего изолированного Docker-контейнера (как отдельная виртуальная машина — чтобы один не сломал другого и не увидел чужих файлов). Гости работают через DeepSeek — это другая языковая модель, которая дешевле Anthropic, но текстовая (не понимает изображения напрямую). Гостям недоступны команды управления ботом и настройки владельца.

**Как попасть в бот.** Новый человек пишет `/start`. Бот видит, что этот ID не в списке, и отправляет владельцу запрос с кнопкой «Одобрить». Когда владелец нажимает кнопку — ID добавляется в разрешённый список, новый пользователь получает сообщение «напиши мне любое сообщение, чтобы начать». В коде бота определён 6-шаговый онбординг (специальный системный промпт-знакомство), но сейчас он по факту не подключён — новые гости сразу попадают в обычный диалог с ботом. Это известный баг, числится в списке на исправление.

Весь этот механизм разделения прописан в файле `src/config.ts` — функция `getUserProfile(userId)` смотрит на ID и возвращает набор настроек конкретного человека: какие папки доступны, какая модель, какие инструменты разрешены, какой системный промпт.

## 3. Что происходит когда пользователь пишет сообщение

Путь сообщения от Telegram до ответа:

1. **Сообщение приходит в бот.** Telegram присылает событие через постоянное соединение (бот сам опрашивает Telegram, это называется «polling»). Файл `src/index.ts` — точка входа — регистрирует обработчики для каждого типа сообщений: текст, голос, фото, документ.

2. **Проверка доступа.** Обработчик `src/handlers/text.ts` первым делом смотрит: входит ли этот ID в список разрешённых пользователей? Если нет — вместо ответа отправляется запрос владельцу на одобрение.

3. **Проверка лимита запросов.** Для владельца работает счётчик — нельзя слать запросы быстрее, чем разрешено настройками. Гостям лимита нет.

4. **Обнаружение темп-смены.** Если сессия уже активна, бот проверяет — это продолжение того же разговора или пользователь перешёл на новую тему? Если тема сменилась — автоматически запускается новая сессия (как `/new`), старая сохраняется.

5. **Подсказка для параллельных задач.** Если сообщение похоже на задачу из нескольких независимых частей («найди 3 кафе», «сравни А и Б», «скачай то, то и то»), бот автоматически добавляет в начало сообщения подсказку: «это задача с несколькими независимыми частями, используй параллельный инструмент». Это нужно потому, что DeepSeek по умолчанию всё делает по очереди, а с подсказкой — запускает части одновременно.

6. **Отправка в Claude / DeepSeek.** Файл `src/session.ts` берёт сообщение и передаёт его в языковую модель. Вместе с сообщением идут: системный промпт (инструкции для модели), список доступных инструментов, история разговора.

7. **Стриминг ответа.** Ответ приходит не целиком, а по частям — как печатает живой человек. Файл `src/handlers/streaming.ts` принимает эти части и постепенно отправляет их в Telegram, обновляя одно сообщение. Промежуточные «размышления» и вызовы инструментов пользователю не показываются — только финальный ответ.

8. **Сохранение в память.** После завершения ответа разговор записывается в транскрипт. Каждые 6 ходов (или при команде `/new`) запускается фоновый анализ: отдельная маленькая модель читает диалог и вытаскивает факты в структурированную память.

## 4. Какой системный промпт получает гость

Системный промпт — это невидимая инструкция, которая идёт перед каждым разговором и говорит модели как себя вести. Функция `buildNewGuestSafetyPrompt` в `src/config.ts` формирует его для каждого гостя.

### Роль

«Ты персональный ИИ-ассистент. Работаешь через Telegram. У тебя есть долгосрочная память, полный набор инструментов, и ты учишься у каждого пользователя в процессе общения. Никогда не говори "как языковая модель я..." — ты просто ассистент, который умеет делать всё».

Бот позиционируется как личный помощник, а не как чат-бот.

### Стиль общения

Промпт требует: писать простым человеческим русским языком, без жаргона и IT-терминов без перевода. «Запушил в репо» → «отправил твои файлы на сервер». Перенимать манеру пользователя: если он пишет коротко — отвечать коротко, если развёрнуто — разворачивать. Запоминать словечки и стиль в файле `profile.md`.

### Память

Промпт описывает три слоя памяти (подробнее — в разделе 6):
- `profile.md` — читать в начале каждого содержательного разговора, обновлять при новых стабильных фактах об этом человеке.
- `graph.json` — структурированные факты и связи. Читать когда пользователь упоминает имена/проекты без контекста.
- `sessions/` — история разговоров по темам. Читать через индекс `topics-index.md`.

Правило: не спрашивать заново то, что уже обсуждали — лучше сначала прочитать память.

### Доступные инструменты

Промпт перечисляет что умеет делать бот (подробнее о каждом — в разделе 7):
- Bash — выполнить команду в терминале. Для гостей работает через Docker-контейнер.
- Read / Write / Edit — читать и писать файлы.
- WebFetch — читать конкретную страницу в интернете. Заменяет WebSearch, которого у DeepSeek нет.
- mcp__send-file — отправить файл пользователю прямо в Telegram.
- mcp__pollinations-image — сгенерировать картинку бесплатно.
- mcp__parallel__run — запустить несколько независимых подзадач одновременно.
- mcp__google-workspace__* — работа с Google Docs, Drive, Sheets, Gmail, Calendar (если пользователь подключил аккаунт).
- mcp__connect-google__connect — подключить Google-аккаунт через авторизацию.

Важная оговорка в промпте: бот не должен говорить «у меня нет инструмента для этого» — нужно сначала попробовать, а потом сообщать о конкретной ошибке.

### Безопасность и запреты

- Рабочая папка — только `/opt/vault/<id>/`. Запрещено читать чужие vault'ы, папки владельца, исходники бота.
- Файлы удалять только с явным «да, удали» от пользователя.
- Нельзя модифицировать конфиг бота (`.env`, исходники, systemd-сервис).

### Ограничения DeepSeek

DeepSeek — текстовая модель, у неё нет понимания изображений (vision). Промпт прямо говорит: «не вызывай инструмент просмотра фото-файлов — вернётся ошибка». Фото от пользователя приходят уже как текстовое описание — бот пропускает их через Gemini перед тем как отдать DeepSeek.

WebSearch тоже недоступен — DeepSeek несовместим с этим инструментом Anthropic. Вместо него — WebFetch на конкретные URL или Bash + curl.

### Ресурсы контейнера

512 МБ оперативной памяти, 1 ядро CPU, своп 512 МБ. Промпт рекомендует перед тяжёлыми операциями проверять свободные ресурсы.

### Публичная страничка

Каждый гость имеет свою публичную папку `${vaultDir}/public/`, доступную в интернете по адресу `https://proboi.site/u/<userId>/`. Можно публиковать туда HTML/CSS/JS файлы и сразу видеть их в браузере.

### Формат ответов

Перед любой задачей с инструментами — сначала короткий анонс (что буду делать, сколько займёт, буду ли запускать параллельные агенты). Потом — работа. Это правило продиктовано промптом: пользователю важно видеть план до того, как что-то начнётся.

## 5. Какой системный промпт получает владелец

Функция `buildOwnerSafetyPrompt` в `src/config.ts`. Общая идеология та же, но разница существенная.

**Что добавлено:**
- Владелец — администратор. Бот должен выполнять его команды по изменению конфига, добавлению пользователей, правке исходников — без лишних вопросов и без ложных отказов («не могу записать файл вне workspace» — это враньё, нужно попробовать и сообщить о реальной ошибке если она есть).
- Добавлен протокол работы с Google Workspace и Gmail — правила чтобы не залить контекст мегабайтами писем при первом же запросе.
- Перечислены десятки интеграций Composio (Notion, Slack, GitHub, Linear, Stripe, Salesforce...) которые можно подключить по запросу.
- Список путей, куда можно писать: `/opt/claude-tg-bot/`, `/root/.claude/`, `/etc/systemd/`, все папки из `ALLOWED_PATHS`.

**Что строже:**
- Удалять файлы — только с явным подтверждением.
- Не запускать `rm -rf` и другие необратимые команды без явного «да».
- После изменения конфига бота — попросить пользователя отправить `/restart` (самому перезапускать нельзя: убьёшь собственный поток ответа).

**Что одинаково с гостевым:**
- Ограничения DeepSeek по vision.
- Правила работы с кодом (Python — всегда через файл, не через bash -c).
- Правила антигаллюцинации: при ошибке инструмента — цитировать дословно, не придумывать причины.
- Анонс плана перед задачей с инструментами.
- Параллельные агенты по умолчанию для задач дольше 10 секунд.

## 6. Где живут данные пользователя

### Личная папка (vault)

Каждый гость имеет личный «шкафчик» на сервере: `/opt/vault/<id>/`. Это его рабочее пространство. Claude видит только эту папку (плюс временную папку `/tmp/telegram-bot/` для входящих файлов).

Структура внутри:
- `memory/<id>/profile.md` — короткий профиль человека: имя, стиль общения, предпочтения. Claude читает его в начале каждого содержательного разговора.
- `memory/<id>/graph.json` — структурированная память в виде узлов и связей. Например: `[person] Андрей → works_on → [project] Откликер`. Это основа долгосрочной памяти.
- `memory/<id>/sessions/` — история разговоров по темам, в виде текстовых файлов с датой и темой в названии.
- `memory/<id>/topics-index.md` — оглавление: какие темы когда обсуждались.
- `skills/` — обучающие инструкции, которые пользователь сам задаёт. Например: «когда я говорю X — делай Y». Claude проверяет эту папку перед каждым запросом.
- `tools/` — скрипты, которые Claude сам написал и сохранил для повторного использования.
- `public/` — папка, содержимое которой доступно по ссылке `https://proboi.site/u/<id>/`.
- `inbox/` — сюда падают входящие фото и документы от пользователя.

### Файл сессии

`/tmp/claude-telegram-session-<id>.json` — список идентификаторов разговоров. Используется командой `/resume` чтобы продолжить предыдущий диалог.

### Что Claude видит, что — нет

Claude (в контейнере гостя) видит только свой vault и `/tmp/telegram-bot/`. Он не видит vault других пользователей, не видит исходники бота, не видит конфиги владельца. Это обеспечивается как промптом (инструкция «не читай чужие папки»), так и физическими ограничениями контейнера.

## 7. Инструменты-помощники (MCP)

MCP-сервер — это отдельная маленькая программа, которая запускается рядом с ботом и даёт Claude доступ к чему-то новому. Claude обращается к ней через специальные вызовы с именем вида `mcp__<название>__<действие>`. Бот перехватывает эти вызовы и делает нужное (отправляет файл, показывает кнопки, запускает задачи).

Какие инструменты сейчас подключены (файл `mcp-config.ts` + `src/mcp-filter.ts`):

**Для всех пользователей:**

- **ask-user** (`mcp__ask-user__*`) — показывает пользователю кнопки в Telegram и ждёт нажатия. Claude использует это когда нужно уточнить выбор: «выбери вариант А или Б».

- **send-file** (`mcp__send-file__*`) — отправляет файл прямо в Telegram-чат пользователя. Claude создаёт файл (Excel, PDF, изображение, любой другой) и вызывает этот инструмент — файл появляется в диалоге.

- **pollinations-image** (`mcp__pollinations-image__*`) — бесплатная генерация изображений через сервис Pollinations (модель Flux). Результат сохраняется во временную папку, откуда send-file его забирает.

- **parallel** (`mcp__parallel__run`) — запускает несколько независимых подзадач одновременно, вместо последовательного выполнения. Например: найти три кафе в трёх параллельных поисках вместо трёх последовательных. Каждая подзадача — это отдельный вызов языковой модели.

- **connect-google** (`mcp__connect-google__connect`) — инициирует подключение Google-аккаунта. Claude вызывает этот инструмент, бот показывает OAuth-кнопку, пользователь нажимает и авторизует доступ.

- **google-workspace** (`mcp__google-workspace__*`) — после подключения аккаунта открывает доступ к Google Docs, Drive, Sheets, Gmail, Calendar. Инструментов там больше сотни. Работает через платформу Composio (управляемый OAuth, не нужен GCP).

**Только для владельца:**

- **openrouter-image** (`mcp__openrouter-image__*`) — платная генерация изображений через OpenRouter (другие модели, более качественные результаты). Требует API-ключ.

**Только для гостей с включённым контейнером:**

- **container** (`mcp__container__Bash`) — выполняет shell-команды внутри Docker-контейнера гостя. Заменяет стандартный Bash, который для гостей отключён. Через этот инструмент можно запустить Python, установить пакет, скачать файл — всё это происходит в изолированной среде.

## 8. Контейнеры (Docker) для гостей

Каждый гость работает внутри своего Docker-контейнера — это как отдельная виртуальная машина со своей файловой системой и ресурсами. Один гость не может случайно залезть в данные другого или перегрузить сервер — контейнер ограничен: 512 МБ памяти, 1 ядро CPU.

Контейнер создаётся при первом сообщении гостя — автоматически, без участия пользователя. После долгого простоя (15 минут) контейнер «засыпает» (docker pause) чтобы не тратить RAM. После суток бездействия — останавливается полностью. При следующем сообщении поднимается заново за несколько секунд.

В контейнере предустановлено: Bun, Node.js 20, Python 3, git, sqlite3, nginx, Calibre (для конвертации книг), pandoc, build-essential. Это позволяет гостю запускать код, ставить пакеты, делать сайты — без установки чего-либо вручную.

Папка vault гостя монтируется в контейнер по тому же абсолютному пути. Это значит: файл созданный внутри контейнера через Bash и файл прочитанный снаружи через Read — это один и тот же файл на диске. Нет двух разных файловых систем.

Всё это управляется через файлы `src/containers/manager.ts` (жизненный цикл: создать, запустить, поставить на паузу, остановить) и `src/containers/spec.ts` (параметры запуска контейнера).

## 9. Метеринг — счётчик токенов

Токены — единица измерения «сколько текста» обработала языковая модель. Каждый запрос стоит денег: входящие токены (то, что модель прочитала) и исходящие (то, что написала).

Бот записывает каждый запрос в SQLite-базу (`metering.sqlite` в корне репозитория). Файл `src/metering.ts` — весь код этого счётчика. Пишется: ID пользователя, модель, источник запроса, количество токенов, посчитанная стоимость в долларах.

Источники запросов:
- `bot-deepseek` — все обычные сессии (и гостей, и владельца — все сейчас на DeepSeek)
- `bot-anthropic` — резерв на случай возврата на Anthropic API; сейчас не используется
- `bot-openrouter` — запросы через OpenRouter (изображения через Gemini, vision-описание фото)
- `bot-openai-whisper` — транскрипция голосовых сообщений

Это нужно для будущего биллинга: когда появятся тарифы, счётчик уже будет знать кто сколько потратил.

**Известные пропуски.** Есть несколько случаев когда токены сейчас не считаются: когда разговор прерывается через ask-user (кнопки), когда пользователь нажимает `/stop`, и когда работает фоновый анализ памяти. Это известные баги, они задокументированы как приоритет на исправление.

## 10. Анализатор памяти

После каждых 6 ходов диалога (или при команде `/new`) бот запускает в фоне отдельный лёгкий запрос к модели с задачей: прочитать этот диалог и вытащить из него факты. Сейчас для всех (и владельца, и гостей) это `deepseek-chat` — он же используется в основной сессии, но уходит как отдельный короткий вызов.

Анализатор (`src/memory/analyzer.ts`) получает транскрипт разговора и список уже известных сущностей из `graph.json`. Модель возвращает структурированный JSON: какие узлы добавить, какие обновить, как они связаны. Например из разговора про покупку ноутбука анализатор создаст узел `[purchase] MacBook Pro M4` со статусом «куплен».

Типы узлов: person, place, trip, project, goal, event, purchase, preference, health, infra (серверный артефакт), topic, fact. Каждый факт записывается один раз, дубли проверяются семантически (чтобы «Шэньчжэнь» и «Shenzhen» не стали двумя разными узлами).

Эта память используется в следующих разговорах: Claude читает граф когда пользователь упоминает что-то без контекста — «помнишь тот проект?», «мы же обсуждали...» — и может ответить конкретно, не переспрашивая.

## 11. Дашборд

На порту 3848 сервера работает встроенный веб-сервер (`src/dashboard-server.ts`). Он показывает страницу с токенами, расходами и ресурсами контейнера. Открывается из бота как Telegram Mini App (всплывает прямо в мессенджере, без перехода в браузер).

Что видно:
- Входящие и исходящие токены — за сегодня и за всё время
- Стоимость в долларах
- Ресурсы контейнера (RAM, CPU, диск)
- Для владельца — сводная таблица по всем пользователям

Аутентификация встроена: страница проверяет подпись Telegram (каждый Mini App получает подписанный токен от Telegram), принимает только данные не старше 24 часов.

## 12. Деплой и боль с бинарником

Прод-сервер: `proboi-bot`, IP 89.167.125.175, бот называется `@proboiAI_bot`. Всё выкатывается через `rsync` (синхронизация файлов) + `bun install` + `systemctl restart`. Никогда не синхронизируется файл `.env` — у каждого сервера свои секретные ключи.

Есть одна регулярная боль: на сервере работает Ubuntu (glibc), а пакетный менеджер Bun иногда устанавливает нативный бинарник Claude CLI для другой системы (musl). После каждого `bun install` нужно вручную заменить этот бинарник на правильную версию, иначе каждый запрос к Claude падает с ошибкой, хотя бот при этом выглядит «живым» — системный сервис активен, но запросы не проходят. Симптом: в логах сервера ошибка «native binary not found». Инструкция по исправлению есть в `CLAUDE.md`.

## 13. Что сейчас может сломаться (известные грабли)

По состоянию на 2026-05-08 (из аудита в `memory/project_knowledge_graph.md`):

**Промпты врут о возможностях (критические):**
- Промпт гостю говорит «фото падают в `/tmp/telegram-bot/`», но на самом деле они кладутся в `${vaultDir}/inbox/`. Если гость спросит «где моё фото?» — Claude ищет не там.
- Промпт обещает инструмент `mcp__connect-google__connect`, но он закомментирован в конфиге и не работает.
- Промпт обещает `mcp__parallel__run`, но parallel-MCP не активирован в конфиге (это не задеплоено на прод).
- Онбординг-промпт определён в коде, но нигде не вызывается — маркер `[ONBOARDING_COMPLETE]` не убирается из чата.
- Промпт говорит гостю «у тебя есть Bash», но он отключён для пользователей с контейнером (там только `mcp__container__Bash`).
- Промпт говорит «есть WebSearch», но он заблокирован для всех гостей — DeepSeek его не поддерживает.

**Метеринг (счётчик токенов) — починено в коммите `be46bdc`, ждёт деплоя:**
- Раньше токены не записывались при прерывании через кнопки ask-user, при `/stop`, и при фоновом анализе памяти. Теперь запись идёт во всех ветках выхода из цикла.
- Для запросов прерванных рано (ask-user/stop до финального события) фоллбек берёт usage из последнего промежуточного assistant-события — то есть учитывается хотя бы последний ход модели до прерывания.
- Анализатор памяти у владельца теперь корректно записывается как `deepseek-chat` (а не как несуществующий haiku).

**Параллельный MCP:**
- `parallel_mcp/server.ts` запускается с рабочей папкой из переменной окружения `TELEGRAM_PARALLEL_CWD`. Для подзадач гостя эта переменная может оказаться неправильной — подзадачи запустятся не в папке гостя, а в дефолтной. Это значит файлы созданные параллельными агентами могут попасть куда не надо или вовсе потеряться.

**Прочее:**
- `openrouter.ts` — часть кода блокирует весь сервер пока идёт запрос (синхронный вызов). При долгих операциях другие пользователи могут ждать дольше обычного.
- `src/fast-path.ts` — написан код для быстрого ответа на простые вопросы в обход Claude CLI (экономия 10-15 секунд), но он не подключён и не задеплоен.

## 14. Где что лежит — карта папок

```
claude-tg-bot/
├── src/                    — всё основное: бот, логика, обработчики
│   ├── handlers/           — обработчики каждого типа сообщений (текст, фото, голос, документы)
│   ├── containers/         — Docker-контейнеры для гостей
│   ├── memory/             — анализатор памяти, граф, транскрипты
│   ├── engines/            — прямые вызовы к DeepSeek и OpenRouter
│   └── templates/          — лендинг и дашборд (HTML)
│
├── ask_user_mcp/           — инструмент «показать кнопки в Telegram»
├── send_file_mcp/          — инструмент «отправить файл пользователю»
├── pollinations_mcp/       — инструмент «сгенерировать картинку бесплатно»
├── openrouter_image_mcp/   — инструмент «сгенерировать картинку платно» (только owner)
├── connect_google_mcp/     — инструмент «начать OAuth для Google»
├── parallel_mcp/           — инструмент «запустить подзадачи параллельно»
│
├── memory/                 — память самого Claude Code (не пользователей!):
│                             граф знаний о проекте, заметки между сессиями
├── graphify-input/         — seed-файлы для построения графа знаний
├── graphify-out/           — визуализация графа (HTML + JSON)
│
├── guide/                  — картинки для онбординга пользователей
├── system/                 — users.json (реестр пользователей)
├── scripts/                — вспомогательные скрипты
│
├── mcp-config.ts           — какие инструменты подключены (не в git, у каждого сервера своя копия)
├── mcp-config.example.ts   — шаблон для новых серверов
├── Dockerfile.user         — образ Docker для гостевых контейнеров
├── metering.sqlite         — база данных токенов (не в git, создаётся автоматически)
└── .env                    — секреты и настройки (не в git, у каждого сервера своя копия)
```

`memory/` в корне — это память Claude Code (то есть моя, как ИИ-агента, который пишет этот документ). Там хранятся заметки о проекте между сессиями работы. Это отдельно от памяти пользователей, которая живёт в `/opt/vault/<id>/memory/`.
