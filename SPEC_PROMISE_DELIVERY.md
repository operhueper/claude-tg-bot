## Архитектурные принципы

1. **Контейнер = дом пользователя.** Всё что делает бот — внутри. Хост не задействуем.
2. **Vault персистентен.** `/opt/vault/<id>/` переживает рестарт контейнера. Скиллы, проекты, данные — там.
3. **Daemon-runner — для всего долгоживущего.** Любой постоянный сервис (бот, scheduler, веб) — через `.daemons.yaml`. PID 1 (Go надсмотрщик) уже умеет revive, healing, problem channel.
4. **MCP — для интерактивного.** ask-user (кнопки), send-file (файлы юзеру), connect-google (OAuth), parallel (фан-аут). Не пишем новые MCP без необходимости.
5. **Безопасность не понижается.** Все фиксы 2026-05-08 (sandbox user, lxcfs, iptables guest-net, cap-drop, pids-limit) сохраняются как инвариант.
6. **Промпт = UI.** Если модель не знает что у неё есть тулза — её нет. Каждая фича = промпт-блок с триггерами и примером.

## Безопасность: что сохраняем как инвариант

Из [SECURITY_AUDIT_REPORT.md](SECURITY_AUDIT_REPORT.md), всё это **не трогаем** при добавлении фич:

- `--user=sandbox` (uid не 0) → нет `/proc/1/root` escape
- `--cap-drop=ALL --no-new-privileges` → нет privesc
- `--read-only` root FS, `tmpfs /tmp` → не пишем в /
- `--pids-limit=128`, `--memory=512m`, `--ulimit=nproc` → нет fork-bomb / OOM
- iptables DROP на `172.17.0.1:3848/3847/22` из `claude-guest-net` → нет dashboard pivot
- `lxcfs` маунты `/proc/meminfo` etc → юзер видит лимиты, не хост
- env whitelist (`buildGuestBaseEnv`) → токены бота не утекают в subprocess
- `seccomp=docker-default`, AppArmor enforce → 56 заблокированных сислов

Все новые фичи **должны вписываться в эти границы**. Если фиче нужно расширение — пишем в спеке отдельным абзацем «изменение security границы» и согласуем.

### Этап 2. Sandbox-образ v2 «всё для PDF/Excel/PPTX/OCR/видео из коробки» (2 часа, **пересборка образа**)

**Цель.** Чтобы первая команда «сделай отчёт PDF» не тратила 90 секунд на `apt install poppler && pip install ...`. Предустановить в образ типовые тулзы.

**Изменения:**

В [containers/Dockerfile.user](containers/Dockerfile.user) добавить:

```dockerfile
# Document tooling
RUN apt-get update && apt-get install -y --no-install-recommends \
    poppler-utils \           # pdftotext, pdfimages
    pandoc \                  # universal markup converter
    wkhtmltopdf \             # html → pdf
    libreoffice-impress \     # pptx → pdf, presentations
    libreoffice-calc \        # xlsx ↔ csv ↔ pdf
    imagemagick \             # convert images
    ffmpeg \                  # audio/video
    tesseract-ocr \           # OCR
    tesseract-ocr-rus \       # russian OCR
    && rm -rf /var/lib/apt/lists/*

# Python data/doc stack
RUN pip install --no-cache-dir \
    openpyxl pandas numpy \
    python-pptx python-docx \
    pillow reportlab pypdf2 pdfplumber \
    requests beautifulsoup4 lxml \
    yt-dlp                    # youtube/video download

# Node tooling
RUN npm install -g \
    puppeteer-core \          # headless Chrome via system chromium
    sharp                     # fast image processing

# Headless browser (для скриншотов / pdf из веб-страниц)
RUN apt-get update && apt-get install -y --no-install-recommends \
    chromium \
    && rm -rf /var/lib/apt/lists/*
```

**Размер.** Текущий образ ~600 MB. Новый ~1.5-2 GB. Это нормально — пересборка раз в недели.

**Тест:** docker build, запустить контейнер вручную, проверить `pdftotext`, `pandoc`, `wkhtmltopdf`, `tesseract`, `ffmpeg`, `python -c "import pptx, openpyxl, pdfplumber"`.

**Деплой:**
1. `docker build -t claude-user-sandbox:latest -f containers/Dockerfile.user .` на проде
2. **Старые контейнеры остаются на старом образе пока не перезапустятся.** Это ок — текущие гости в работе не прерываются.
3. При следующем create — новые получают новый образ.
4. Опционально: `docker ps --filter label=claude-bot-user -q | xargs docker restart` — насильно поднять всех на новом образе. **Только с подтверждением** — у двух юзеров крутятся персональные боты, рестарт = downtime их сервисов.

**Риск:** средний. Если apt-get что-то ломает (incompatible deps) — откат на старый image tag.

**Безопасность:** новые тулзы запускаются в том же sandbox, тех же лимитах. Imagemagick исторически имел RCE через специально сформированные SVG — добавим `policy.xml` отключающий опасные кодеки.

### Этап 4. Cron-из-чата «расписание и триггеры» (3 часа, **новый код**)

**Цель.** Закрыть обещание «расписание и триггеры — настраивается в чате одной фразой».

**Архитектурное решение.** Не делаем хостовый scheduler (хост = не наш слой, а гости все на одном хосте → нужен per-user crontab → лишняя сложность). Делаем **scheduler внутри контейнера через daemon-runner**.

**Дизайн:**

1. В образе уже будет cron (после этапа 2 ставим `cron` пакет).
2. В `.daemons.yaml` добавляем «системный демон» `bot-scheduler` (Go или Python), который:
   - читает `/workspace/.schedule.yaml` (расписания пользователя)
   - выполняет команды по cron-выражениям
   - результат пишет в `/workspace/.schedule-runs/<timestamp>.log`
   - при ошибке — пишет в Telegram через bot API (используя токен из vault, если пользователь настроил собственного «notify» бота)
3. **Промпт-блок** учит модель: «когда юзер просит "каждое утро в 9 присылай X" — модифицируй `/workspace/.schedule.yaml`, не пиши свой костыль».

**Формат `.schedule.yaml`:**
```yaml
schedules:
  - name: morning_news
    cron: "0 9 * * *"
    cmd: ["python3", "/workspace/scripts/news.py"]
    notify: telegram   # шлёт результат пользователю через bot API
  - name: weekly_report
    cron: "0 18 * * fri"
    cmd: ["bash", "/workspace/scripts/weekly.sh"]
```

**Уведомления юзеру.** Самое тонкое место — *как scheduler шлёт сообщение пользователю*. Варианты:

- **A. Через основной @proboiAI_bot** (наш токен). Scheduler внутри контейнера НЕ имеет доступа к токену (env whitelist). Решение — exposed-эндпоинт в боте: scheduler стучит на `http://172.17.0.1:3849/notify` (новый порт, только из claude-guest-net, с валидацией userId по контейнеру). Бот шлёт сообщение в чат.
- **B. Через личный бот пользователя.** Если юзер уже создал свой telegram-бот (есть у 2 юзеров), его токен в его vault, scheduler шлёт через него напрямую. Сложно предполагать что у каждого есть.

**Выбор:** A — универсальнее. Создаём `notify-bridge` в `src/dashboard-server.ts` (или отдельный сервер на 3849), валидация по source IP контейнера → userId mapping.

**Безопасность:**
- Эндпоинт принимает только из `172.17.0.0/16` (iptables)
- Тело запроса содержит userId, бэкенд проверяет что source IP принадлежит контейнеру этого userId (через `containerManager.getIp(userId)`)
- Rate limit per userId
- Нет attachments (только текст) — иначе можно DoS-ить ботом

**Тест:**
1. На тестовом аккаунте: «напоминай мне каждые 5 минут что я красавчик» → scheduler пишет в `.schedule.yaml`, через 5 мин приходит сообщение.
2. «Удали все напоминания» → файл очищается.
3. Перезапустить контейнер → расписания продолжают работать (daemon-runner поднимает scheduler автоматически).

**Деплой:** новый код в репо + новый системный демон в `.daemons.yaml` шаблоне для гостей. Существующим гостям — миграционный скрипт добавляет демон в их `.daemons.yaml`.

**Риск:** средний (новый сетевой эндпоинт, новый исполняемый компонент). Тестировать на одном аккаунте до общего раскатывания.

### Этап 5. Фоновые задачи «работаю в фоне, пришлю когда готово» (2 часа)

**Цель.** Долгие задачи (анализ 100 документов, генерация большого отчёта) не блокируют чат на 5 минут. Юзер пишет «посчитай X», бот говорит «работаю, пришлю как готово», юзер уходит, через N минут приходит результат.

**Дизайн:**

1. Модель в контейнере по триггеру «это долгая задача» запускает скрипт через `nohup` или backgrounded process в контейнере.
2. Скрипт пишет результат в `/workspace/.tasks/<task_id>.json` (status, result, error).
3. Тот же scheduler-демон (этап 4) опрашивает `/workspace/.tasks/` каждые N секунд, при `status=done` или `error` — шлёт уведомление через notify-bridge.
4. Промпт-блок: «если задача займёт > 60 секунд — запусти в фоне, скажи юзеру "работаю, пришлю как готово"».

**Безопасность.**
- pids-limit=128 уже есть → не более 128 одновременных процессов (включая фоновые)
- mem 512 MB — фоновая задача не выйдет за лимит контейнера
- Если контейнер убивают (idle > N минут) — фон тоже умирает. Это ОК для v1, но не идеально. v2: контейнер не уходит в pause если есть active task.

**Тест:** «скачай и проанализируй 50 youtube-видео» → бот отвечает «работаю», через несколько минут — результат.

**Деплой:** требует этапа 4 (нужен notify-bridge). Делаем сразу после.

**Риск:** низкий, переиспользуем инфру этапа 4.

**Тест:** «сделай мне бота-таймер, который посылает напоминания» → пользователь даёт токен → через 2 минуты получает работающий бот.

**Деплой:** только промпт + skill-файл, без нового кода в основном боте.

**Риск:** низкий, инфра уже есть, добавляем UX.

## Открытые вопросы для пользователя

1. **Этап 2 — пересобирать ли образ сейчас или отложить?** Образ +1 GB, скачивается раз. Альтернатива — оставить минимальный, ставить тулзы по запросу (медленнее в первый раз, бесплатно).
2. **Этап 2 — рестартовать ли все живые контейнеры?** У двух юзеров крутятся постоянные боты — их рестарт = их боты на ~10 секунд молчат. Можно подождать естественного рестарта.
3. **Этап 4 — какой язык для scheduler-демона?** Go (как daemon-runner) или Python (быстрее писать)? Я бы сделал Go для консистентности.
4. **Этап 4 — notify-bridge — отдельный порт или встроить в dashboard-server?** Я бы вынес в отдельный (3849), чтобы security boundary было чище.
5. **Этап 6 — лимит по количеству ботов на юзера?** Скажем, 3 одновременных. Иначе юзер нагенерит 50 ботов и убьёт pids-limit. (Сейчас лимит 128 PIDs на контейнер — этого хватит на 5-10 простых ботов.)

## Готовность к старту

Этап 1 могу начать сразу после твоего «ок этап 1». Это **самый безопасный** фикс — только промпт, ломаться нечему. Юзеры сразу почувствуют разницу: модель перестанет говорить «не могу выполнить bash», начнёт реально исполнять команды.

После этапа 1 — отчёт «вот что починил, вот тест, ок на этап 2?»
