# Роадмап: функции Claude Code для Proboi

**Дата:** 2026-05-11  
**Проект:** Telegram-бот «Proboi» (claude-tg-bot)  
**Стек:** Bun + TypeScript, ~3300 строк, прод на proboi-bot (89.167.125.175)

---

## Контекст

Бот «Proboi» предоставляет доступ к Claude Code и DeepSeek через Telegram. Гости работают в изолированных Docker-контейнерах с DeepSeek API, owner — через Claude API. Текущий UX ближе к «умному чату», чем к полноценному агентному окружению: нет видимого прогресса задач, нет защиты от неожиданных деструктивных действий, контекст обрывается при переполнении.

Этот документ описывает пять функций из Claude Code, которые сделают бот значительно более предсказуемым, надёжным и удобным для реальной работы.

---

## Функция 1. Plan Mode — предварительный план перед выполнением

### Что это

Перед тем как выполнять задачу с потенциально деструктивными или необратимыми шагами (удаление файлов, запись в БД, отправка запросов во внешние сервисы, установка пакетов), Claude формирует текстовый план и показывает его пользователю. Выполнение не начинается, пока пользователь не нажмёт одну из трёх кнопок: «Выполнить», «Отменить» или «Уточнить».

### Зачем пользователю

Сейчас Claude может начать делать что-то значимое сразу после команды. Пользователь в Telegram не видит код до его выполнения — только результат. Это создаёт риск неожиданных действий, особенно у гостей, которые работают в Docker-контейнерах с реальными данными в `/opt/vault/{userId}/`. Plan Mode даёт секунду на оценку и возможность скорректировать задачу до того, как что-то произошло.

### Технический подход

**Файлы, которые затрагиваются:**

- `src/session.ts` — перехват и остановка стрима при обнаружении маркера
- `src/handlers/text.ts` — запуск с флагом plan mode, отображение плана
- `src/handlers/callback.ts` — обработка нажатий на кнопки плана
- `src/config.ts` — промпт-инструкция, настройка режима per-profile

**Шаг 1. Промпт-инструкция для LLM**

В системный промпт (в `getUserProfile()` в `config.ts`) добавляется блок:

```
Перед выполнением задачи, которая содержит деструктивные действия
(удаление файлов, запись в базу данных, отправка во внешние сервисы,
установка пакетов, изменение конфигурации), сначала выведи план
в следующем формате и ничего больше не делай:

PLAN_START
Шаг 1: <описание>
Шаг 2: <описание>
...
PLAN_END

После вывода PLAN_END остановись и жди подтверждения пользователя.
Если задача безопасна (чтение файлов, вычисления, объяснения) — выполняй сразу.
```

**Шаг 2. Перехват в session.ts**

В `sendMessageStreaming` добавляется детектор маркера в потоке текста. Когда в ответе встречается `PLAN_START`, стрим приостанавливается — накапливается текст плана до `PLAN_END`. После `PLAN_END` запрос к LLM не продолжается (используем `AbortController`). Частичный контекст (план) сохраняется в сессии как `pendingPlan: string`.

```typescript
// В session.ts добавляется поле в состояние:
interface SessionState {
  pendingPlan?: {
    planText: string;
    originalMessage: string;
    abortedAt: Date;
  };
}
```

**Шаг 3. Отображение плана в text.ts**

После того как стрим прерван с `pendingPlan`, `text.ts` вызывает `ctx.reply()` с текстом плана и тремя inline-кнопками:

```
callback_data: plan_confirm:{userId}
callback_data: plan_cancel:{userId}
callback_data: plan_clarify:{userId}
```

Сообщение хранит `message_id` для последующего редактирования.

**Шаг 4. Обработка в callback.ts**

- `plan_confirm` — берёт `session.pendingPlan.originalMessage`, добавляет в сообщение «(пользователь подтвердил план)», запускает сессию заново с полным контекстом
- `plan_cancel` — очищает `pendingPlan`, редактирует сообщение: «Задача отменена»
- `plan_clarify` — редактирует сообщение: «Опишите, что изменить в плане», переводит пользователя в режим уточнения (следующее текстовое сообщение трактуется как уточнение к исходной задаче)

**Хранение состояния**

`pendingPlan` хранится в сессионном файле `/tmp/claude-telegram-session-${userId}.json`. При `/new` или `/stop` — очищается.

**Граничные случаи**

- Если LLM выводит `PLAN_START` без `PLAN_END` (обрыв) — тайм-аут 30 секунд, затем сброс
- Если пользователь пишет новое сообщение при активном `pendingPlan` — уведомить: «Есть ожидающий план, нажмите кнопку или /new для отмены»
- Plan Mode включается только для профилей, где `planMode: true` в `UserProfile`; у owner по умолчанию включён, у гостей — включается при наличии флага в профиле

### Оценка сложности

**M** — 2-3 рабочих дня.

Основная сложность — надёжный парсинг маркеров в потоке (стрим может разбить `PLAN_START` по частям). Нужно буферизировать вывод и искать маркер по накопленной строке, а не по отдельным чанкам.

### Зависимости

Не зависит от других фич роадмапа. Может быть реализована первой при изменении приоритетов.

## Функция 2. Todo-список — видимый прогресс в реальном времени

### Что это

Пока Claude выполняет многошаговую задачу, пользователь видит отдельное «служебное» сообщение с актуальным списком задач. Каждый пункт имеет один из трёх статусов: ожидает, в работе, выполнен. Список обновляется по мере выполнения.

```
Задачи:
◻ Прочитать файл конфигурации
⏳ Установить зависимости
✅ Проверить версию Node
```

### Зачем пользователю

Сейчас длинные задачи выглядят как «тишина», прерываемая итоговым ответом. Пользователь не знает, завис ли бот или работает. Todo-список даёт прозрачность: понятно, на каком шаге Claude, сколько осталось, нет ли зависания на одном действии.

### Технический подход

**Файлы, которые затрагиваются:**

- `src/handlers/streaming.ts` — парсинг маркеров, управление служебным сообщением
- `src/session.ts` — инструкция для LLM
- `src/config.ts` — промпт-блок

**Шаг 1. Промпт-инструкция для LLM**

В системный промпт добавляется блок:

```
Для многошаговых задач (более 2 шагов) используй специальные маркеры прогресса.
В самом начале ответа объяви все шаги:

TODO_LIST_START
TODO_ITEM:read_config:Прочитать конфигурацию
TODO_ITEM:install_deps:Установить зависимости
TODO_ITEM:check_version:Проверить версию
TODO_LIST_END

Перед каждым шагом выведи:
TODO_START:read_config

После завершения шага:
TODO_DONE:read_config

Маркеры выводи на отдельных строках, без дополнительного текста вокруг.
```

**Шаг 2. Парсинг маркеров в streaming.ts**

`StreamingState` расширяется полями:

```typescript
interface StreamingState {
  // ... существующие поля
  todoMessageId?: number;
  todoItems: Array<{
    id: string;
    label: string;
    status: 'pending' | 'in_progress' | 'done';
  }>;
}
```

В потоке обработки текстовых чанков добавляется `parseTodoMarkers(chunk, state)`. Функция ищет маркеры и:

- При `TODO_LIST_START..END` — создаёт массив `todoItems` со статусом `pending`, отправляет служебное сообщение, сохраняет `todoMessageId`
- При `TODO_START:id` — переводит нужный item в `in_progress`, вызывает `bot.api.editMessageText(todoMessageId, renderTodoList(state.todoItems))`
- При `TODO_DONE:id` — переводит item в `done`, обновляет сообщение
- Маркеры фильтруются из текста, который идёт в основное сообщение пользователю

**Рендер списка:**

```typescript
function renderTodoList(items: TodoItem[]): string {
  const icon = { pending: '◻', in_progress: '⏳', done: '✅' };
  const lines = items.map(i => `${icon[i.status]} ${i.label}`);
  return '<b>Выполняю:</b>\n' + lines.join('\n');
}
```

**Шаг 3. Жизненный цикл служебного сообщения**

- При `done` (завершение стрима) — служебное сообщение удаляется или обновляется финальной строкой «Готово»
- При ошибке или прерывании — обновляется соответствующим статусом
- `StreamingState.maxSegmentId` уже умеет удалять промежуточные сообщения — todo-сообщение должно быть помечено отдельно, чтобы не попасть в удаление текстовых сегментов

**Шаг 4. Буферизация маркеров**

Стрим может разбить маркер по двум чанкам: `TODO_ST` / `ART:id`. Нужен небольшой кольцевой буфер (последние 100 символов) для поиска маркеров на стыке чанков. Это тот же подход, что используется в streaming.ts для обнаружения `<tool_call>` тегов.

### Оценка сложности

**M** — 1-2 рабочих дня.

Основная сложность — надёжный парсинг стрима с буферизацией и синхронизация `editMessageText` (Telegram ограничивает частоту редактирования). Редактировать не чаще раза в секунду — добавить дебаунс.

### Зависимости

Не зависит от других фич. Можно реализовать параллельно с Plan Mode.

## Функция 3. Компакция контекста — автоматическое сжатие при приближении к лимиту

### Что это

Когда история сессии приближается к токеновому лимиту модели, бот автоматически сжимает старые сообщения в краткое резюме. Пользователь получает уведомление «Контекст сжат, продолжаем», а сессия продолжается без обрыва.

### Зачем пользователю

Сейчас при длинных сессиях Claude Code возвращает ошибку превышения контекста, сессия рвётся. Пользователь получает невнятную ошибку и вынужден делать `/new`, теряя все договорённости и промежуточный контекст. Компакция решает эту проблему прозрачно.

### Технический подход

**Файлы, которые затрагиваются:**

- `src/session.ts` — основная логика компакции
- `src/types.ts` — типы для сжатого контекста
- `src/config.ts` — пороги и настройки

**Шаг 1. Мониторинг длины контекста**

В `sendMessageStreaming` перед каждым запросом проверяется оценочная длина контекста. Поскольку session.ts использует Agent SDK (`query()`), прямого доступа к `messages[]` нет — SDK управляет историей через файл сессии `/tmp/claude-telegram-session-${userId}.json`.

Нужно читать этот файл перед каждым запросом и считать суммарный размер в символах (грубая оценка: 1 токен ≈ 4 символа). Для DeepSeek-гостей лимит ~64K токенов, для Claude — 200K.

```typescript
async function estimateContextSize(sessionFile: string): Promise<number> {
  try {
    const data = await fs.readFile(sessionFile, 'utf-8');
    return JSON.parse(data).messages?.reduce(
      (sum: number, m: any) => sum + JSON.stringify(m).length, 0
    ) ?? 0;
  } catch {
    return 0;
  }
}
```

**Шаг 2. Триггер компакции**

Если `estimatedTokens > threshold * modelLimit` (порог 0.8 по умолчанию), запускается `compactSession(userId, profile)`.

Настройки в `config.ts`:

```typescript
compaction: {
  enabled: true,
  threshold: 0.8,          // % лимита для триггера
  keepRecentMessages: 10,  // сколько свежих сообщений не трогать
  summaryModel: profile.model, // DeepSeek для гостей, Claude для owner
}
```

**Шаг 3. Функция compactSession**

```typescript
async function compactSession(userId: number, profile: UserProfile): Promise<void> {
  const sessionFile = `/tmp/claude-telegram-session-${userId}.json`;
  const session = JSON.parse(await fs.readFile(sessionFile, 'utf-8'));
  const messages = session.messages ?? [];

  // Оставляем последние N сообщений нетронутыми
  const keepCount = profile.compaction.keepRecentMessages;
  const toSummarize = messages.slice(0, -keepCount);
  const toKeep = messages.slice(-keepCount);

  if (toSummarize.length === 0) return;

  // Формируем запрос на суммаризацию
  const summaryPrompt = buildSummaryPrompt(toSummarize);
  const summary = await callSummaryModel(summaryPrompt, profile);

  // Заменяем старые сообщения одним summary-блоком
  const summaryMessage = {
    role: 'system',
    content: `[Сжатый контекст предыдущего диалога]\n${summary}`,
    _isCompactionSummary: true,
    _summarizedCount: toSummarize.length,
    _summarizedAt: new Date().toISOString(),
  };

  session.messages = [summaryMessage, ...toKeep];
  await fs.writeFile(sessionFile, JSON.stringify(session, null, 2));
}
```

**Шаг 4. Промпт суммаризации**

```typescript
function buildSummaryPrompt(messages: any[]): string {
  const dialog = messages
    .map(m => `${m.role === 'user' ? 'Пользователь' : 'Ассистент'}: ${
      typeof m.content === 'string' ? m.content : JSON.stringify(m.content)
    }`)
    .join('\n\n');

  return `Сделай краткое резюме следующего диалога. Сохрани:
- ключевые договорённости и решения
- названия файлов, переменных, функций, которые обсуждались
- незавершённые задачи
- важные факты и ограничения

Диалог:
${dialog}

Резюме (не более 2000 слов):`;
}
```

**Шаг 5. Уведомление пользователя**

После успешной компакции `sendMessageStreaming` отправляет в чат:

```
Контекст сессии стал слишком большим. Сжал историю диалога, 
сохранив ключевые договорённости. Продолжаем.
```

**Шаг 6. Хранение метаданных компакции**

В sessionFile добавляется поле:

```json
{
  "messages": [...],
  "compactions": [
    {
      "at": "2026-05-11T10:30:00Z",
      "summarizedMessages": 47,
      "summaryLength": 1823
    }
  ]
}
```

Это помогает при отладке и позволяет отслеживать, сколько раз сессия была сжата.

**Потенциальные проблемы**

- Agent SDK перезаписывает sessionFile при каждом запросе — нужно сжимать файл ДО запуска `query()`, а не в обработчике стрима
- При одновременных запросах (маловероятно в однопользовательском потоке, но возможно) — file-lock или mutex на уровне userId
- Качество суммаризации: DeepSeek может терять детали кода. Тест: 5 реальных сессий до сжатия и после — проверить, понимает ли Claude контекст из summary

### Оценка сложности

**L** — 3-4 рабочих дня.

Технически не сложно, но требует тестирования качества суммаризации на реальных сессиях. Плохое резюме хуже, чем отсутствие резюме. Нужно несколько итераций промпта.

### Зависимости

Независима от других фич. Критически важна — сессии рвутся прямо сейчас. Первый приоритет.

## Функция 4. Память у гостей — persistent memory между сессиями

### Что это

После каждой сессии Claude сохраняет краткое резюме диалога и выученные факты о пользователе в постоянное хранилище. В начале следующей сессии это резюме автоматически добавляется в контекст. Гость возвращается — Claude помнит его проекты, предпочтения, незавершённые задачи.

### Зачем пользователю

Сейчас каждый гость начинает с нуля. После `/new` или перезапуска бота — весь контекст теряется. Память даёт ощущение «помощника, который тебя знает»: не нужно каждый раз объяснять, на каком языке писать код, как называется твой проект, какой стиль предпочитаешь.

### Технический подход

**Файлы, которые затрагиваются:**

- `src/memory/` — новая директория (или расширение существующей, если есть `memory/analyzer.ts`)
- `src/session.ts` — инжект памяти в начало сессии и запись после завершения
- `src/config.ts` — пути к памяти, лимиты
- `src/handlers/commands.ts` — команда `/forget` для сброса памяти

**Структура хранилища**

Память каждого гостя хранится в `/opt/vault/{userId}/memory/`:

```
/opt/vault/{userId}/memory/
  index.json          — оглавление: timestamp, размеры, количество записей
  sessions/
    2026-05-11.json   — резюме сессий за день
    2026-05-10.json
  facts.json          — выученные факты о пользователе (язык, стиль, проекты)
  MEMORY.md           — читаемая версия для инжекта в промпт
```

**Шаг 1. Запись памяти после сессии**

В `session.ts`, в обработчике завершения `sendMessageStreaming` (блок `finally` или `on('done')`):

```typescript
async function saveSessionMemory(userId: number, profile: UserProfile, messages: Message[]): Promise<void> {
  if (profile.isOwner) return; // owner уже имеет свою память через ~/.claude
  if (messages.length < 3) return; // слишком короткая сессия, не сохранять

  const memoryDir = `/opt/vault/${userId}/memory`;
  await fs.mkdir(memoryDir, { recursive: true });

  const summary = await generateSessionSummary(messages, profile);
  const today = new Date().toISOString().slice(0, 10);

  // Дописываем в файл дня
  const dayFile = `${memoryDir}/sessions/${today}.json`;
  const existing = await readJsonSafe(dayFile, { sessions: [] });
  existing.sessions.push({
    at: new Date().toISOString(),
    summary,
    messageCount: messages.length,
  });
  await writeJsonSafe(dayFile, existing);

  // Обновляем MEMORY.md
  await rebuildMemoryMd(userId, memoryDir);
}
```

**Шаг 2. Генерация резюме сессии**

Используется DeepSeek (дешевле, подходит для суммаризации):

```typescript
async function generateSessionSummary(messages: Message[], profile: UserProfile): Promise<string> {
  const prompt = `Проанализируй диалог и выдели:

4. Факты о пользователе (язык, стиль, предпочтения)
```

**Шаг 3. Инжект памяти в начало сессии**

В `sendMessageStreaming`, перед первым запросом в новой сессии (определяется по отсутствию истории в sessionFile):

```typescript
async function injectMemoryContext(userId: number, profile: UserProfile): Promise<string | null> {
  if (profile.isOwner) return null;
  
  const memoryFile = `/opt/vault/${userId}/memory/MEMORY.md`;
  try {
    const content = await fs.readFile(memoryFile, 'utf-8');
    if (content.length < 50) return null; // пустая/минимальная память
    
    return `[Память о пользователе из предыдущих сессий]\n${content}\n[Конец памяти]`;
  } catch {
    return null; // памяти ещё нет
  }
}
```

Если memory context получен — он добавляется как первое system-сообщение в историю сессии.

**Шаг 4. MEMORY.md — читаемое резюме**

```typescript
async function rebuildMemoryMd(userId: number, memoryDir: string): Promise<void> {
  const sessions = await loadRecentSessions(memoryDir, 14); // последние 14 дней
  const facts = await readJsonSafe(`${memoryDir}/facts.json`, {});
  
  const lines = [
    '## Последние сессии\n',
    ...sessions.slice(-5).map(s => `- ${s.at.slice(0, 10)}: ${s.summary}`),
    '\n## Выученные факты\n',
    ...Object.entries(facts).map(([k, v]) => `- ${k}: ${v}`),
  ];
  
  const content = lines.join('\n');
  
  // Лимит 50 KB
  const truncated = content.length > 50_000
    ? content.slice(-50_000)
    : content;
  
  await fs.writeFile(`${memoryDir}/MEMORY.md`, truncated);
}
```

**Шаг 5. Ротация по LRU**

При превышении 50 KB в `MEMORY.md`:
- Сначала удаляются самые старые записи сессий (по дате файла)
- Факты не удаляются, только старые детали перезаписываются при обновлении

Ротация запускается в `rebuildMemoryMd` как проверка размера.

**Шаг 6. Команда /forget**

В `commands.ts` добавляется команда, доступная гостям:

```
/forget — удалить всю память о себе
```

Реализация: удаляет `/opt/vault/{userId}/memory/`, отвечает «Память очищена. Начинаем с чистого листа.»

Команда добавляется в `baseCommands` (для гостей).

**Шаг 7. Команда /memory**

```
/memory — показать, что бот помнит о вас
```

Выводит содержимое `MEMORY.md` пользователю. Если файла нет — «Пока ничего не сохранено».

### Оценка сложности

**L** — 3-4 рабочих дня.

Технически прямолинейно, но требует аккуратности в нескольких местах:
- Определение «конца сессии» — когда записывать? (лучший момент: по `/new` или через 30 минут неактивности)
- Качество summary: DeepSeek должен выделять действительно полезные факты, а не пересказывать
- Privacy: пользователь должен легко удалить свою память

### Зависимости

Не зависит от других фич. Желательно реализовывать после отладки компакции контекста (Фича 3) — похожая логика суммаризации, можно переиспользовать промпты.

## Функция 5. Прерывание с перенаправлением — плавный контроль выполнения

### Что это

Сейчас `!стоп` (восклицательный знак в начале сообщения) прерывает сессию через `checkInterrupt()`. После прерывания контекст теряется, и пользователь должен заново формулировать задачу. Новое поведение: `!<текст>` прерывает текущее выполнение, сохраняет контекст и немедленно начинает новый запрос с уточнением.

Пример:
- Claude устанавливает 50 зависимостей
- Пользователь пишет `!стоп, используй только стандартную библиотеку`
- Claude останавливается, понимает контекст прерывания, начинает заново с новым ограничением

### Зачем пользователю

Текущий `!стоп` — грубый инструмент. Иногда пользователь хочет не отменить задачу, а скорректировать подход на лету. Без перенаправления приходится делать `/new` и объяснять всё заново. Прерывание с перенаправлением даёт управление без потери контекста.

### Технический подход

**Файлы, которые затрагиваются:**

- `src/utils.ts` — `checkInterrupt()`, расширение для сохранения контекста
- `src/session.ts` — `savePartialContext()`, возобновление с redirect-сообщением
- `src/handlers/text.ts` — определение `!<текст>` (не просто `!стоп`)

**Шаг 1. Расширение checkInterrupt**

Текущая `checkInterrupt(text, userId)` проверяет, начинается ли текст с `!`. Расширяем: если начинается с `!` и после `!` есть текст (не только «стоп» или пустота) — это redirect-прерывание.

```typescript
export function checkInterrupt(text: string, userId: number): InterruptResult {
  if (!text.startsWith('!')) return { isInterrupt: false };
  
  const content = text.slice(1).trim();
  const isStopOnly = ['стоп', 'stop', ''].includes(content.toLowerCase());
  
  return {
    isInterrupt: true,
    isRedirect: !isStopOnly,
    redirectMessage: isStopOnly ? null : content,
  };
}

interface InterruptResult {
  isInterrupt: boolean;
  isRedirect?: boolean;
  redirectMessage?: string | null;
}
```

**Шаг 2. Сохранение контекста при прерывании**

В `session.ts` добавляется `savePartialContext()`:

```typescript
async function savePartialContext(userId: number, partialResponse: string): Promise<void> {
  const sessionFile = `/tmp/claude-telegram-session-${userId}.json`;
  try {
    const session = JSON.parse(await fs.readFile(sessionFile, 'utf-8'));
    // Добавляем частичный ответ Claude как assistant-сообщение
    // Это сохраняет контекст до момента прерывания
    if (partialResponse.length > 50) {
      session.messages = session.messages ?? [];
      session.messages.push({
        role: 'assistant',
        content: `[Выполнение прервано пользователем]\n${partialResponse}`,
        _isPartialResponse: true,
        _interruptedAt: new Date().toISOString(),
      });
      await fs.writeFile(sessionFile, JSON.stringify(session, null, 2));
    }
  } catch {
    // Если файл недоступен — продолжаем без сохранения
  }
}
```

**Шаг 3. Обработка redirect в sendMessageStreaming**

Сейчас при обнаружении `!` в потоке, `checkInterrupt` устанавливает флаг и стрим прерывается. Расширяем:

```typescript
// В sendMessageStreaming, в обработчике AbortSignal
if (abortResult.isRedirect && abortResult.redirectMessage) {
  // Сохраняем контекст до прерывания
  await savePartialContext(userId, accumulatedText);
  
  // Формируем redirect-запрос
  const redirectPrompt = `Пользователь прервал выполнение и дал новое указание: "${abortResult.redirectMessage}". 
Учти контекст предыдущего выполнения (оно было прервано) и продолжи с учётом нового указания.`;
  
  // Запускаем новый запрос с redirect-промптом
  // Небольшая задержка для того, чтобы abort полностью завершился
  setTimeout(() => {
    sendMessageStreaming(redirectPrompt, userId, profile, ctx, statusCallback);
  }, 500);
}
```

**Шаг 4. Уведомление пользователя**

При redirect-прерывании (до запуска нового запроса):

```
Прерываю выполнение. Применяю ваше уточнение...
```

**Граничные случаи**

- Abort race: Claude Code subprocess иногда выходит с code 1 при abort — это уже подавляется в `session.ts`. Redirect добавляется только если abort завершился без ошибки
- Если `accumulatedText` пустой (Claude ничего не успел вывести) — перенаправление работает как обычный новый запрос
- Вложенные прерывания: если пользователь шлёт `!` во время redirect-запроса — обрабатывается стандартно (ещё одно прерывание)
- Таймаут 500 мс перед новым запросом — нужен для гарантии, что abort полностью отработал. Если SDK abort races станут проблемой — увеличить до 1-2 секунд

**Сложности с SDK abort**

Agent SDK (`query()` через subprocess) не предоставляет надёжного способа получить частичный ответ после abort. `accumulatedText` накапливается в обработчике стрима до момента прерывания — это лучшее, что доступно без модификации SDK. На практике: если Claude успел вывести несколько сотен символов до `!` — контекст сохранится; если прерывание пришло в первые секунды — контекст будет пустым.

### Оценка сложности

**M-L** — 2-3 рабочих дня.

Основная сложность — abort race conditions с SDK. Надёжность зависит от того, насколько предсказуем abort в конкретной версии Claude CLI. Требует тестирования на реальных длинных задачах.

### Зависимости

Зависит от наличия надёжного механизма abort — то есть желательно реализовывать после хорошего понимания abort behavior в текущей версии SDK (`@anthropic-ai/claude-agent-sdk` 0.2.x). Не зависит от других фич роадмапа.

## Приоритизированный порядок реализации

### 1. Компакция контекста (критично)

**Почему первая:** сессии рвутся прямо сейчас. Это единственная фича из пяти, которая устраняет существующий баг, а не добавляет новую функциональность. Пользователи теряют контекст при длинных задачах — это блокирующая проблема для продуктивной работы.

**Когда откладывать:** если все активные пользователи работают только с короткими задачами (до 10-15 сообщений), срочность снижается. Но по мере роста нагрузки проблема станет хуже.

### 2. Todo-список (UX — видимость прогресса)

**Почему вторая:** самая быстрая в реализации (1-2 дня) при значительном UX-эффекте. Убирает «чёрный ящик» — пользователь перестаёт гадать, работает ли бот. Особенно важно для гостей, которые платят за время.

**Почему не первая:** не устраняет существующую боль, а улучшает опыт. Компакция важнее.

### 3. Plan Mode (безопасность)

**Почему третья:** защищает от деструктивных сюрпризов, но только если Claude правильно определяет «деструктивные» задачи. Качество определения зависит от промпта — нужна итерация. Ставим после Todo-списка, потому что к тому времени будет больше понимания, как LLM следует промпт-инструкциям в потоке.

**Нюанс:** для гостей в Docker-контейнерах риски деструктивных действий ограничены sandbox'ом. Plan Mode более критичен для owner, у которого доступ к `/opt` и конфигурации сервера.

### 4. Прерывание с перенаправлением (UX — контроль)

**Почему четвёртая:** улучшает опыт управления, но требует хорошего понимания abort behavior в текущей версии SDK. Лучше реализовывать после того, как компакция и todo прошли тестирование — они дадут понимание того, насколько предсказуем стрим и abort.

**Риск:** если SDK abort работает нестабильно — redirect может терять контекст. В таком случае фича деградирует до «просто нового запроса с уточнением», что всё равно лучше нынешнего состояния.

### 5. Память у гостей (retention)

**Почему последняя:** требует наибольшего времени и оказывает эффект на удержание пользователей, а не на решение текущих болей. Является стратегической функцией — имеет смысл реализовывать, когда основные UX-проблемы решены и есть аудитория, которую нужно удерживать.

**Дополнительный аргумент:** логика суммаризации из компакции контекста (Фича 3) напрямую переиспользуется здесь. Реализуя компакцию первой, мы получаем часть фундамента для памяти бесплатно.

## Технические риски

**Качество LLM-инструкций.** Все пять фич зависят от того, насколько точно LLM следует специальным маркерам (`PLAN_START`, `TODO_ITEM`, и т.д.). DeepSeek и Claude ведут себя по-разному. Нужно тестировать каждую инструкцию на обеих моделях до деплоя.

**Парсинг в потоке.** Маркеры могут разбиваться по чанкам. Для каждой фичи нужен буфер последних N символов. Использовать паттерн уже применённый в `streaming.ts`.

**Частота редактирования Telegram.** `editMessageText` лимитирован: не более 1 редактирования в секунду на сообщение (практически — ~0.5). Todo-список и Plan Mode обновляют сообщения; нужен дебаунс.

**Файловые race conditions.** Компакция и память читают/пишут файлы сессии. При параллельных операциях (маловероятно, но возможно при быстрых сообщениях) нужен mutex per userId. Реализуется через Map<userId, Promise> с цепочкой промисов.

**Размер памяти.** 50 KB на гостя при 100 активных гостях = 5 MB. Для `/opt/vault/` это не проблема. Но нужна ротация, чтобы один активный пользователь не накопил гигабайты старых резюме.
