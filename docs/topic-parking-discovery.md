# Авто-парковка контекста при смене тем — research для следующей сессии

**Дата:** 2026-05-14
**Статус:** только research, никакого кода ещё не написано.
**Цель документа:** дать будущей сессии полную базу, чтобы можно было комплексно подумать и решить — что и в каком объёме делать.

---

## 1. Проблема

Пользователи с СДВГ (живой пример — Ксения, userId `893951298`) прыгают с темы на тему. У бота сейчас **один сквозной диалог на юзера**:

- Метаданные сессий: `/tmp/claude-telegram-session-${userId}.json` (5 последних `sessionId`)
- Транскрипт: `memory/{userId}/transcripts/{sessionId}.jsonl` (по строке на турн, без обрезки)
- Реальную историю держит **SDK через `resume: sessionId`** — мы её напрямую не контролируем

Когда юзер переключается «код → маркетинг → фото → опять код», SDK честно тащит всю историю → модель путается, цепляет мусор из соседних тем, в худшем случае извиняется и пишет «я тебя дезинформировал дважды».

---

## 2. Что выяснилось (5 research-агентов, сводно)

### 2.1 Память — частично уже есть

Источник: `src/session.ts`, `src/memory/`, `src/handlers/streaming.ts`

| Что | Где | Состояние |
|---|---|---|
| Метаданные сессий | `/tmp/claude-telegram-session-${userId}.json` | пишется `saveSession()` в `session.ts:1256-1291`, MAX_SESSIONS=5 |
| Турн-лог | `memory/{userId}/transcripts/{sessionId}.jsonl` | пишется `TranscriptRecorder`, **без ротации** |
| Per-user knowledge graph | `memory/{userId}/graph.json` | есть `src/memory/graph.ts` с GraphStore, atomic write |
| Контекст-инжекция | `buildMemoryContext(graph, goals, ...)` | вызывается на старте новой сессии, top-20 нод |
| Периодический анализ | `session.ts:1207-1218` | каждые 6 турнов, экстракция в граф |
| Компрессия | — | вырезана 2026-05-14 (`6a2f66c`, ломала `StreamingState`) |

**Известные риски памяти:**
1. Транскрипт растёт без обрезки (5000 турнов = гигабайты)
2. Если SDK падает между `session_id` и `result`, `TranscriptRecorder` может не инициализироваться → молчаливая потеря
3. `maxTurns` — только SDK-уровень, локально не проверяется

### 2.2 Графы — `graphify-out/` не подходит, `src/memory/graph.ts` подходит

| Что | Назначение | Подходит для тредов? |
|---|---|---|
| `graphify-out/` | статический индекс кода (1147 нод, AST-extraction) | **НЕТ** — это индекс артефактов кода, не runtime-данные юзеров |
| `memory/project_knowledge_graph.md` | ручной markdown-лог сессий владельца | НЕТ — текст, не структура |
| `src/memory/graph.ts` + `memory/{userId}/graph.json` | per-user knowledge graph, atomic write | **ДА** — сюда добавляем коллекцию `threads` рядом с `nodes` |

**Утверждённое решение:** хранить треды внутри существующего `memory/{userId}/graph.json`, в **отдельной коллекции** (один файл, два логических раздела: `nodes` и `threads`). Реюзаем atomic-write код.

### 2.3 Ксения — outlier, не репрезентативна

userId `893951298`, 131 сообщение за 2026-05-07…05-13. Три железобетонных кейса переключения:

**Кейс 1 (2026-05-07 → 05-08):** Notion-стратегия Reels → 14ч паузы → печать 6 фоток на A4 PDF.
Никаких связок, началось с резкого `Где?` → `Ты ебобо? у тебя все есть` → `Сделай`. Бот гадал, она матом возвращала.

**Кейс 2 (2026-05-13):** Уишань (Китай, велосипеды) → стоматологии Шэньчжэнь → reply-quote назад к Уишань.
Абрупт без связок. Бот извинялся: «Ксюх, прости реально 🙈... я тебя дезинформировал дважды».

**Кейс 3 (2026-05-13):** Кроссовки на Poizon → «Помнишь проект откликер?» (2ч паузы).
Явный recall-маркер, бот отреагировал правильно.

**Паттерн:**
- Переключения **абруптны**, **без связок** («кстати/а ещё» у неё в словаре нет)
- Возвращается через `Помнишь...` или Telegram reply-quote (цитата старого сообщения бота)
- Когда бот теряет контекст — эскалирует: короткие «Где?» «А?» → мат → reply-quote-якорь
- Длинная пауза (>2ч) часто совпадает со сменой темы

### 2.4 Контрольная группа — Ксения outlier

Топ-2 активных гостя помимо Ксении (анонимизировано):

**Гоша (`946882308`), 358 сообщений за неделю:**
- Глубокое копание одного проекта (бот для 3 барбершопов: зарплаты, штрафы, план из yclients)
- Median gap = 157s, всего 10 пауз >2ч
- **Использует «теперь» 26 раз и «давай» 51 раз внутри одной темы** — главный риск false-positive
- Использует валидные recall-маркеры: «помнишь» 6 раз, «вернёмся» 2 раза
- 1 абруптный switch без связок: `Давай сделаем сайт с презентацией бота` после долгой отладки

**Пользователь `299753724`, 169 сообщений:**
- Тоже один проект (бот «Пятница 2.0» для обучения), линейный
- Median gap = 208s, 8 пауз >2ч
- «помнишь» 2 раза, «кстати/а ещё/стоп» = 0 раз
- Один textbook recall: `Помнишь мы делали бота. Сейчас мы этапе создания тестов` после >24ч паузы

**Артём (`5615267984`), 161 сообщение:** смешанный, туристика + бот-работа, 3-й лучший кандидат в тест.

**Вердикт:** Ксения — outlier. Контрольные юзеры — deep-drillers с переключением **на границе дней**, а не в течение разговора. Классификатор надо тюнить **тоньше** чем «только под Ксению», иначе Гоша получит 80 ложных форков за неделю.

### 2.5 Edge cases (золотая жила agent #5)

Из 1175 сообщений 9 активных юзеров за неделю:

| Категория | Примеры | Частота |
|---|---|---|
| **B. Reply-to-quote** | `[В ответ на сообщение от ...: «...»]\n\n<user text>` | **81 случай, ~7% траффика** |
| **E. Burst-merge** | 2-3 user-сообщения подряд без ответа бота, dt<30s | **146 случаев, ~12% траффика** |
| **A. Photo+caption clusters** | 6 фото вин с микро-вариациями caption «Какой рейтинг этого вина?» | 12 случаев |
| **C. Bot-self-confusion** (lagging!) | `прости`, `извини`, `не понял`, `запутался`, `косячу` | ~14 случаев |
| **D. Short messages** | `Где?`, `Сделай`, `Ну?`, `Спасибо`, `Дальше`, `разрешаю` | много |
| **F. Multi-day return без маркера** | `что там про тарифы?` после 45ч паузы | 3 случая |

**Главные находки:**

1. **Reply-quote — сильнейший сигнал в датасете**, сильнее любого LLM-классификатора. Просто вытащить `«...»` и считать топик-анкором.
   - НО фильтровать META-цитаты: `«Извини, я только что перезапустился»` — это не топик, это шум.

2. **Burst-merge обязателен**: если юзер шлёт 3 сообщения за 20 секунд без ответа бота — это **одна логическая реплика**, не три темы. Классифицировать ПОСЛЕ склейки.

3. **Короткие подтверждения (`ок/хорош/давай/дальше/разрешаю/ага`) при dt<60s** → forced `same_topic` правилом, классификатор не дёргать.

4. **Photo-burst (`[Photos:` маркер)** = всегда одна тема, независимо от caption-вариаций.

5. **Многодневные возвраты без «помнишь»** (типа «что там про тарифы?») — нужен noun-overlap со старой историей. Это уже **per-user долговременная память**, не runtime-классификатор.

**Скрытые ловушки:**
- `Нет ответа от модели` / `No response from Claude.` (~20 случаев) НЕ считать историей — иначе следующая реплика юзера выглядит как topic-switch, хотя это frustration-retry на ту же тему
- Опечатки в caption (`ретинг` вместо `рейтинг`) — нужна typo-tolerant similarity
- `разрешаю` — выглядит как continuation-лексикон, но это echo permission-кнопки бота → META-фильтр
- Owner cross-talk про самого бота («Посмотри что у Ксюши случилось») — meta-уровень, не пользовательская тема (если для owner вообще включать парковку)

**Leading vs lagging:**
- **Leading** (можно действовать в реальном времени): reply-quote, recall-маркеры, паузы, noun-overlap, short-affirm continuation, photo-burst, burst-merge
- **Lagging** (видно только постфактум): бот извинился, мат от юзера, короткие fragmented сообщения → это **eval/test signal**, не runtime input

## 3. Утверждённые решения

| Вопрос | Ответ |
|---|---|
| Движок классификатора | **DeepSeek** (других у гостей нет) |
| UX при форке | «🔀 Заметил новую тему, переключаюсь. Вернёмся к старой когда будешь готов» |
| UX при возврате | **Авто, с кратким подтверждением**: «↩️ Продолжаем про X, остановились на: ...» |
| Гранулярность решения | **Гибрид**: одиночное при сильном сигнале (пауза >2ч / recall-маркер); иначе окно из 2-3 сообщений |
| Хранилище тредов | Внутри `src/memory/graph.ts`, в отдельной коллекции `threads` (один файл, два логических раздела) |

## 4. Предложенная архитектура v1 (черновик, не финал)

Слоистая система, **не «один LLM на каждое сообщение»**:

```
[Сообщение] →
  [Слой 1: пре-фильтр правилами, без LLM]
    ├─ burst-merge: <30s, нет ответа бота → склеить
    ├─ reply-quote extract: вытащить [В ответ на ... «X»]
    ├─ short-affirm + dt<60s → forced SAME (выход)
    ├─ photo-burst → SAME (выход)
    └─ META-фильтр: отсечь "перезапустился"/"нет ответа" из истории
  ↓
  [Слой 2: сильные сигналы для авто-форка]
    ├─ recall-маркеры ("помнишь", "вернёмся к", "тот X") → классификатор + поиск треда
    ├─ пауза >2ч + новый noun-phrase opener → классификатор
    └─ reply-quote к турну >1ч → классификатор + анкор-трейд
  ↓
  [Слой 3: DeepSeek-классификатор]
    Input: склеенная реплика + reply-quote-анкор + сводка 2-3 прошлых турнов + список заголовков активных тредов
    Output: SAME | NEW | RETURN(thread_id)
    Промпт: "сомневаешься — SAME" (консерватизм по Гоше)
  ↓
  [Слой 4: действие]
    NEW    → создать тред в graph.threads, парковка старого с auto-summary, новый sessionId, UX-уведомление
    RETURN → resume старого sessionId, UX "↩️ продолжаем про X, остановились на: ..."
    SAME   → ничего, продолжаем
```

### Структура данных треда (черновик)

В `memory/{userId}/graph.json`:

```json
{
  "nodes": [...],
  "edges": [...],
  "threads": [
    {
      "id": "uuid",
      "title": "Бот для барбершопов: зарплаты",
      "sessionId": "sdk-session-uuid",
      "createdAt": "2026-05-07T13:43:00Z",
      "lastActiveAt": "2026-05-13T13:47:00Z",
      "summary": "Юзер настраивает 3 филиала, спорят про время отправки в 22:00 vs 22:15",
      "status": "active",
      "anchorNouns": ["зарплата", "филиал", "Калужская", "Коммунарка"]
    }
  ],
  "currentThreadId": "..."
}
```

`anchorNouns` — для F-кейса (multi-day return по noun-overlap).

### Почему это «не один классификатор»

- **80%+ сообщений Гоши не пойдут в LLM вообще** (слой 1 их закроет короткими аффирмами и burst-merge)
- **Кейсы Ксении пойдут в слой 2** (пауза >2ч / абрупт opener) и решатся
- DeepSeek дёргается реже → дешевле, быстрее, меньше латентности на типичный турн

## 5. Открытые вопросы — для завтрашнего решения

### Q1. Объём v1

| Вариант | Что входит | Срок |
|---|---|---|
| **MIN** | Слои 1+3+4, без recall-маркеров и multi-day return, ручные `/threads` `/resume_thread` | ~1 неделя |
| **MID** (рекомендован) | + recall-маркеры, + reply-quote anchor, + сильные временные сигналы. Покрывает 3 кейса Ксении и реальные returns Гоши | ~2 недели |
| **MAX** | + noun-overlap для >24ч, + offline eval framework | ~3-4 недели |

### Q2. Роллаут

| Вариант | Плюсы | Минусы |
|---|---|---|
| Все гости сразу | быстрый сбор данных | Гоша первым словит ложные форки если слой 1 проредит плохо |
| Фича-флаг на 3 теста (Ксения + Гоша + 299753724) (рекомендован) | контролируемый blast radius, разные паттерны | каждого надо предупредить |
| Сначала офлайн-прогон на 1175 сообщениях | нулевой риск, метрики до прода | дольше на 1-2 дня |

### Q3. Судьба существующего `/resume`

| Вариант | Плюсы | Минусы |
|---|---|---|
| `/resume` показывает треды вместо сессий | один UX | надо переделать UI команды |
| `/resume` рядом с `/threads`, старый flow не трогать (рекомендован) | минимальный риск поломки | две похожих команды |
| Депрекейтим `/resume`, оставляем `/threads` | чистый ABI | ломает привычку владельца |

### Q4. Открытые design-вопросы вне 3-х choice-блоков

- **Как делать summary при парковке треда?** Отдельный DeepSeek-вызов после форка? Или брать последние N турнов как есть и не суммировать (просто текстовый префикс на resume)?
- **Что показывать при возврате?** Только заголовок треда + 1-2 предложения summary? Или полный turn-by-turn пересказ?
- **Как ограничить число активных тредов на юзера?** Гоша за неделю мог наплодить 5-10 тредов (3 филиала × 2-3 направления). Лимит? Авто-архив через N дней без активности?
- **Что делать с уже накопленной историей при включении фичи?** Все существующие сессии → один thread "разное"? Или попробовать LLM-кластеризацию прошлых сообщений на треды?
- **Что считать "META-сообщением" при reply-quote?** Жёсткий блоклист («перезапустился», «нет ответа», «бесплатный для тебя»)? Регексп? Или классификатор?
- **Как не дёргать классификатор для owner?** Владелец многозадачный по природе работы (бот сам себя дебажит, ползает по серверам) — для него парковка может быть только мешать. Фича-флаг owner-off?

## 6. Что НЕ делаем в v1 (defer)

- Бот-confusion detection (агент 5, категория C) — это **lagging signal**, использовать только как eval set, не как runtime input
- Rage-vs-drill disambiguation на коротких `Где?` `Ну?` — требует session-level state (есть ли pending deliverable, был ли error в прошлом ответе)
- Multi-day implicit anaphora через сессии без noun-overlap (`что там про тарифы?` когда тарифы не упоминались ни в одной сохранённой сессии) — нужна персистентная топик-память, отдельная задача
- Meta-system noise (`разрешаю`, эхо UI-кнопок) — нужно знание о состоянии бот-UI
- Owner cross-talk про сам бот — нужен owner-mode carve-out, не v1
